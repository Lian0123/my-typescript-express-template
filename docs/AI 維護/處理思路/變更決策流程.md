# 變更決策流程

## 先分類

- 行為修正：先建立可重現案例，修正最靠近根因的層，加入回歸測試。
- 新功能：先定義輸入、輸出、錯誤與資料生命週期，再沿既有分層加入元件。
- 重構：維持外部行為，先有測試保護，再分小步變更。
- 依賴／配置：先讀 release notes 與現有相容性，驗證 build、啟動與資料遷移。
- 文件：以可驗證來源更新，避免把計畫寫成現況。

## 影響面檢查

1. API 或 Socket contract 是否改變？同步 DTO、AO、Swagger、測試。
2. 資料模型是否改變？同步 Entity、migration、repository、測試；不得只依賴 synchronize。
3. DI constructor 是否改變？同步 `src/container.ts` 與測試替身。
4. 事件 payload／名稱是否改變？同步 producer、consumer、失敗策略與測試。
5. 環境變數是否改變？同步 `.env.example`、compose 與安裝配置索引。
6. 是否跨越帳號／角色與聊天邊界？讀兩邊索引再動手。

## 最小變更原則

- 修根因，不以 catch-all 或關閉檢查掩蓋錯誤。
- 不在同一變更內混入無關重命名、格式化與套件升級。
- 保持公開 contract 相容；破壞性變更需明確版本策略。
- 新增抽象前確認至少有兩個真實使用點或明確測試價值。
- 不能驗證的假設寫入任務紀錄，不寫入「專案事實」。
