# 開發規範索引

## 分層規則

- Controller：解析／驗證輸入、宣告 API contract、呼叫 service、轉換 AO；不直接查資料庫。
- Service：業務規則、跨 repository 協調、事件觸發；不依賴 Express Request。
- Repository：資料存取與持久化細節；不處理 HTTP 回應。
- DTO/AO/BO/PO/Entity：維持各層資料邊界，欄位變更需追蹤整條資料流。
- Container：唯一 production 組裝點；constructor dependency 改變時同步 binding 與測試替身。

## TypeScript 與命名

- 沿用現有 decorator metadata、CommonJS 與路徑風格，除非任務明確做現代化遷移。
- 公開方法與 contract 使用明確型別；避免新增 `any`、非必要 type assertion 與忽略錯誤註解。
- 非同步呼叫必須 await 或明確處理 promise；連線與 channel 要考慮關閉與失敗。
- 檔名使用既有 `<domain>.<layer>.ts`；測試為 `.spec.ts`，E2E 為 `.e2e.spec.ts`。

## API、資料與安全

- 所有外部輸入走 class-validator DTO；分頁與 ID 也要驗證及轉型。
- 錯誤透過既有 `serviceError`／`ErrorHandler` contract，不回傳內部 stack。
- schema 變更建立 migration 並驗證 rollback；不要只依賴 `synchronize`。
- 事件 payload 視為跨元件 contract，變更需相容策略。
- 日誌不可含密碼、token、完整個資或未遮罩 payload。

## Git 與範圍

採 Conventional Commits（repository 已配置 Commitizen）。一個變更集中一個目的；提交前檢查 `git diff`、測試結果、生成檔與 living docs。
