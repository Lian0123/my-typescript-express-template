# 依 AI 文件維護專案

使用 living docs 導航，但以程式碼與執行結果作最終事實來源。

## 工作流

1. 只讀 `docs/AI 維護/入口.md`，依任務路由選一份任務索引與一份功能索引；不要預先讀完整文件庫。
2. 使用 `rg` 驗證索引中的路徑、符號、contract 與測試。文件若漂移，將同步修正列入本次範圍。
3. 以 `任務處理/任務工作流.md` 定義目標、非目標、證據、影響與驗證；未指定但可安全推導的細節採既有模式。
4. 檢查 API／Socket、資料模型、DI、事件、環境、安全與相容性影響。跨功能域時才讀第二份功能索引。
5. 實作最小完整變更。修 bug 先建立失敗證據；新功能先定義 contract 與不變條件；不要混入無關清理。
6. 新增或更新回歸測試，依 `測試與驗證索引.md` 執行與風險相稱的 build、test、lint、整合檢查。
7. 用 git diff 列出變更檔，再依 `文件生成索引.md` 同步文件：
   - 行為／資料流／路徑 → 功能索引
   - 架構／能力邊界 → 專案事實
   - 新增、解除或改級風險 → 風險與技術債
   - 環境／指令 → 安裝配置或測試索引
   - 對使用者或維護者有意義的交付 → CHANGELOG
8. 只更新本次已重新核對文件的 `verified` 日期，執行 `npm run docs:verify`，最後檢查 diff。

## 必守規則

- 不把文件當成實作證據；衝突時先驗證再修文件。
- 不為小改動全量重寫文件，也不把過程筆記塞進重點記憶。
- 不宣稱未執行的測試通過；列出指令、結果與限制。
- 不提交密鑰、實際 `.env`、個資或未遮罩 payload。
- 新功能必須能由入口在兩次連結內定位；若無合適功能頁，建立一份精簡索引並連回功能總索引與入口。

## 完成條件

程式行為、測試、索引、風險記錄與 CHANGELOG 彼此一致；所有本地文件連結通過驗證；下一個小模型不需掃描整個 repository 即可找到變更入口與驗證方式。
