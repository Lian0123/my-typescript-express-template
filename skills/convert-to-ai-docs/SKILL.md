# 轉換為 AI 維護文件

將來源整理為導航與事實，不把原文機械搬運進文件庫。

## 工作流

1. 讀取 repository 根目錄 `AGENTS.md` 與 `docs/AI 維護/入口.md`；若不存在，依本 skill 的「最小架構」建立。
2. 盤點輸入來源及其日期、適用版本、可信度。先讀目錄與標題，再只讀與專案現況相關的區段。
3. 用目前程式碼、設定、測試與可執行結果驗證每項主張。無法證實的內容標記「待驗證／提案」，不得寫入專案事實。
4. 將資訊分流：
   - 穩定架構與能力邊界 → `重點記憶/專案事實.md`
   - 會影響決策的缺陷或限制 → `重點記憶/風險與技術債.md`
   - 責任、入口、資料流、搜尋詞 → `快速定位/*索引.md`
   - 重複操作與品質門檻 → 對應根層索引或 `任務處理/`
   - 外部資料 → `外部參考/`，保留 URL、日期、版本與摘要
   - 已交付的重要變更 → `異動記錄/CHANGELOG.md`
5. 就地合併既有權威文件。避免新增同義頁、複製長程式碼、行號定位、對話歷程與暫時推論。
6. 更新入口的任務路由；確保新增資訊可在入口後兩次連結內抵達。
7. 依 `docs/AI 維護/文件生成索引.md` 補 metadata，只更新真正重新驗證過的日期。
8. 執行 `npm run docs:verify`，再人工抽查連結、事實來源與繁體中文可讀性。

## 最小架構

必須包含入口、讀取策略、專案事實、風險、功能總索引、任務工作流、安裝配置、開發規範、測試驗證、文件生成、日誌索引、CHANGELOG 與外部參考索引。依真實功能域增加快速定位頁，不預建空白功能文件。

## 轉換判準

- 一個事實只有一個權威位置，其他地方以相對連結引用。
- 使用穩定檔案路徑與符號名稱；內容超過單一任務所需時拆成一層子文件。
- 明確區分「現況、限制、提案、歷史」。
- 保留專案原始技術詞與 symbol，說明使用繁體中文。
- 不寫入密鑰、個資、內部連線值、未授權資料或大段受版權保護的外部文字。

若輸入與現況衝突，保留目前可驗證事實；只有衝突會影響後續決策時，才在風險或歷史記錄衝突與來源。
